# 9. 파운데이션 모델(foundation model)과 가상 세포(virtual cell)

**파운데이션 모델(Foundation Model)**은 대규모 생물학 자료로 사전학습한 뒤 여러 하위 과제에 전이할 수 있도록 설계된 모델 계열을 뜻한다. 이 절에서는 서열·구조·기능 예측을 GEM의 확정 입력이 아니라 출처와 불확실성을 가진 **후보**로 다룬다. **가상 세포(Virtual Cell)**는 서열·오믹스·GEM·동역학·실험을 연결하려는 연구 의제이며, 구성 요소의 존재만으로 실시간 전세포 예측 또는 폐쇄 루프 검증이 성립하지는 않는다.

## 9.0 사전학습 모델과 과제 특화 모델의 역할을 구분한다

이 장의 §3 RF, §4 GNN, §6 AMN은 대체로 특정 GEM·세포주·예측 과제에 맞춘 모델이다. 반면 파운데이션 모델은 넓은 입력 자료에서 사전학습(pre-training)한 표현을 특정 과제의 추가 학습 또는 미세조정(fine-tuning)에 사용하는 접근이다. 의학의 전문과 비유는 입력 범위와 재사용 가능성을 설명할 뿐, 모델이 생물학적 기전이나 임상적 타당성을 자동으로 판단한다는 뜻은 아니다. 따라서 EC 번호나 결합 예측은 [Chapter 5](../chapter-5/README.md)의 반응 후보를 우선순위화할 수는 있어도, GPR·반응·방향성을 확정하지 않는다.

## 9.1 AlphaFold3: 단백질-리간드 구조 예측

**[AlphaFold3](https://deepmind.google/technologies/alphafold/)**와 같은 구조 예측 결과는 모델·서버 버전, 입력 엔터티, 신뢰도 지표와 벤치마크 정의를 함께 확인해야 한다. GEM 재구성에서는 orphan enzyme의 구조 또는 활성 부위에 관한 후보, GPR 검토를 위한 보조 정보, 효소-기질 상호작용을 시험할 가설로 사용할 수 있다. 그러나 구조 또는 결합의 예측만으로 반응 방향성, 기질 특이성, GPR 규칙을 확정할 수는 없다. 각 후보는 문헌·주석·질량수지 검사를 거쳐 [Chapter 5](../chapter-5/README.md)의 큐레이션 결정과 독립 실험 검토로 이어져야 한다.

## 9.2 ESM-3: 단백질 언어 모델

**[ESM-3](https://github.com/evolutionaryscale/esm)**와 같은 단백질 언어 모델의 입력 모달리티와 허용된 출력은 설치한 모델 카드와 릴리스에서 확인해야 한다. GEM 재구성에서는 기능 미상 유전자의 EC 후보, 변이체 또는 설계 후보를 제안하는 데 사용할 수 있다. 출력마다 입력 서열·모델 버전·신뢰도·후보 사용 목적을 기록하고, 반응 커버리지의 추가 또는 효소 기능의 수용은 독립 근거가 확인된 뒤에만 수행한다.

## 9.3 효소 동역학 Foundation Model: GECKO 3.0, HIT-EC

효소 제약 모델(enzyme-constrained model, ecModel)은 단순화한 1:1 효소-반응 대응에서 $$v_i \leq k_{cat,i} E_i$$와 같은 용량 제약을 사용한다. 여기서 $$v_i$$는 반응 플럭스, $$k_{cat,i}$$는 효소의 회전수, $$E_i$$는 같은 생물량 기준으로 환산한 활성 효소량이다. 이 식을 적용하려면 시간 단위, 효소량의 기준, 반응-효소 매핑, 복합체·동질효소 처리와 배양 조건이 서로 맞아야 한다.

단위 계산을 위한 장난감 예를 보자. $$k_{cat}=10\ \text{s}^{-1}$$, $$E=2\ \text{mmol·gDW}^{-1}$$이고 활성 효소 한 단위가 반응 한 단위에 대응한다고 가정하면 $$v\leq20\ \text{mmol·gDW}^{-1}\text{s}^{-1}$$이다. 시간 단위를 바꾸면 $$20\times3{,}600=72{,}000\ \text{mmol·gDW}^{-1}\text{h}^{-1}$$이다. 예를 들어 $$k_{cat}=8\text{--}12\ \text{s}^{-1}$$, $$E=1.5\text{--}2.5\ \text{mmol·gDW}^{-1}$$의 입력 범위를 단순 곱으로 전파하면 용량 범위는 $$12\text{--}30\ \text{mmol·gDW}^{-1}\text{s}^{-1}$$, 즉 $$43{,}200\text{--}108{,}000\ \text{mmol·gDW}^{-1}\text{h}^{-1}$$이다. 이 값은 단위 확인을 위한 교육용 수치이며, 실제 단백질 풀·복합체·조건 제약을 가진 모델의 예측값이 아니다.

예측 $$k_{cat}$$ 또는 EC 분류기를 ecModel 구축에 연결할 때는 도구 이름이나 단일 성능 수치만으로 채택하지 않는다. 다음처럼 모델·자료·판정 기준을 분리해 기록한다. 현재 이 절의 GECKO, DLKcat, ECMpy, HIT-EC 관련 릴리스와 벤치마크 수치는 근거 장부 검증 전이므로 결과값으로 사용하지 않는다.

| 검토 대상 | 최소 기록 | GEM에서 허용되는 결론 | 아직 허용되지 않는 결론 |
|---|---|---|---|
| $$k_{cat}$$ 예측기 | 모델·가중치 버전, 학습/평가 분할, 입력 기질 표현, 단위, 불확실성 | 측정 우선순위 또는 용량 제약의 후보 | 실제 $$k_{cat}$$ 값의 확정 또는 모든 반응의 자동 치환 |
| ecModel 구축 도구 | 공식 릴리스, 입력 GEM·프로테옴·데이터베이스, 매핑/QC, solver 상태 | 재현 가능한 실행 계약의 후보 | 의사코드만으로 실행 가능성 또는 성능 향상 주장 |
| EC 분류기 | 데이터셋, 계층 정의, split, 평균 방식, 기준선, 해당 subset | 주석 검토 후보 | 특정 F1을 다른 종·데이터셋의 반응 커버리지 개선으로 일반화 |

{% hint style="warning" %}
**개념적 API · 실행 건너뜀:** 아래 코드는 효소 제약 모델 구축의 입력과 출력 관계를 설명하는 의사코드이다. 이 저장소에는 `yeast-GEM.xml`과 `proteomics.csv`가 없으며, 실제 ECMpy 릴리스의 공개 API가 아래 이름과 같다고 보장하지 않는다. 따라서 이 절에서는 실행 결과를 제시하지 않는다. 실행하려면 공식 릴리스·라이선스, GEM 및 프로테옴 checksum, 예측기와 $$k_{cat}$$ 데이터베이스 버전, 환경·solver, 매핑 및 QC 결과를 하나의 manifest로 고정해야 한다.
{% endhint %}

```python
# PSEUDOCODE ONLY — ECMpy 2.0을 이용한 ecModel 자동 구성의 개념적 API
from ecmpy import build_ec_model
import cobra

model_yeast = cobra.io.read_sbml_model("yeast-GEM.xml")
ec_model = build_ec_model(
    model_yeast,
    kcat_predictor='dlkcat',            # 'dlkcat' | 'turnup' | 'autopacmen'
    proteomics_data='proteomics.csv',   # 선택적 프로테오믹스 통합
    organism='yeast'
)
solution = ec_model.optimize()
print(f"Growth rate: {solution.objective_value}")
```

| 실행 manifest 항목 | 이 절의 상태 | 실행 전 필요한 확인 |
|---|---|---|
| GEM·프로테옴 입력 | 없음 | 경로, checksum, 생물종·배양 조건 |
| 패키지·예측기·$$k_{cat}$$ 자료 | 고정되지 않음 | 공식 릴리스, 라이선스, 버전, 입력 schema |
| solver와 QC | 실행하지 않음 | status, 질량수지, 단위·매핑 점검, 제약 위반 |
| 출력 | 없음 | 목적함수, 경계조건, tolerance와 함께 저장 |

## 9.4 Self-Driving Lab과 디지털 트윈: Virtual Cell을 향한 통합

**자율 연구소(Self-Driving Laboratory, SDL)**는 설계·실험·측정·학습의 일부를 자동화하여 반복하는 시스템을 가리킨다. 어떤 시스템을 SDL로 부르더라도 물리적 측정, 작동(action), 자율성의 경계, 전향적 검증 범위를 따로 기록해야 한다. 이를 “GEM이 스스로 가설을 만들고 세포를 편집해 임상적으로 검증하는 완전자율 실험실”과 동일시해서는 안 된다.

**디지털 트윈(Digital Twin, DT)**은 측정값으로 갱신되는 공정 또는 개체의 계산 모델을 뜻한다. 단순 시뮬레이터, 상태 추정기, 폐쇄 루프 제어기는 서로 다른 주장 수준이다. 한 공정의 개선 결과를 다른 제품·세포주·스케일로 옮기려면 별도 검증이 필요하다.

**가상 세포(Virtual Cell)**는 서열·구조·오믹스·GEM·동역학·실험을 연결하려는 장기 연구 의제이다. 이 절에서는 “구성 요소가 존재한다”와 “end-to-end 자율 순환이 실증되었다”를 서로 다른 검증 주장으로 취급한다. 후자를 주장하려면 시간·공간 범위, 상태 갱신, 제어 행동, 전향적 검증과 독립 재현의 근거를 함께 제시해야 한다.

다음 표는 성숙도를 이름이 아니라 관찰 가능한 증거로 분류하는 틀이다. 특정 시스템의 빈칸을 채우려면 해당 시스템의 실행·측정 자료가 필요하며, 빈칸을 성숙도 달성으로 해석하지 않는다.

| 구분 | 측정으로 상태 갱신 | 물리적 작동 | 전향적 검증 | 독립 재현 | 이 절에서의 판정 규칙 |
|---|---|---|---|---|---|
| 시뮬레이터 | 선택적 또는 없음 | 없음 | 모델 적합도와 별도 | 별도 확인 | 예측만으로 twin 또는 SDL이라 부르지 않는다 |
| 상태 추정기 | 필요 | 없음일 수 있음 | 예측 시점 이후 자료로 확인 | 별도 확인 | 갱신 주기와 오차를 기록한다 |
| 폐쇄 루프 제어 | 필요 | 필요 | 잠긴 목표·안전 경계에서 확인 | 별도 확인 | action과 outcome의 인과 범위를 기록한다 |
| SDL | 과제별로 명시 | 과제별로 명시 | 과제별 전향 결과 필요 | 별도 확인 | 자동화 범위와 사람의 승인 단계를 기록한다 |
| 가상 세포 연구 의제 | 구성 요소별로 확인 | 구성 요소별로 확인 | 통합 범위별로 확인 | 별도 확인 | 구성 요소 목록만으로 통합 완성을 판정하지 않는다 |

### Virtual Cell을 이 장의 다른 절과 겹쳐 보기

가상 세포라는 큰 그림 안에서 이 장의 다른 절들이 맡는 후보 역할을 정리하면 다음과 같다. 이 표는 여러 하위 모델을 연결하는 연구 의제를 설명할 뿐, 통합된 검증 결과를 뜻하지 않는다.

| Virtual Cell의 구성 요소 | 이 장에서 다룬 대응 기술 |
|:---|:---|
| 유전자→단백질 구조·기능 | §9.1 AlphaFold3, §9.2 ESM-3 |
| 반응 네트워크의 화학량론적 뼈대 | [Chapter 2](../chapter-2/README.md)·[Chapter 3](../chapter-3/README.md)의 GEM 재구성 |
| 효소 용량·동역학 제약 | §9.3 GECKO, DLKcat |
| 빠른 상태 예측(정상상태를 넘어선 시간 변화) | §6의 서로게이트, dFBA |
| 조절·설계 의사결정 | §7의 RL 기반 균주 설계 |
| 문헌·주석 지식의 자동 정비 | §8의 LLM 큐레이션 |
| 예측 신뢰도 평가 | §10의 불확실성 정량화 |

{% hint style="warning" %}
**해석상의 주의:** 구성 요소 목록은 통합 시스템의 검증 근거가 아니다. 단일 시스템을 주장하려면 각 연결의 입력·출력, 시간·공간 범위, 상태 갱신, 제어 행동, 전향적 검증과 독립 재현을 같은 계약 안에서 제시해야 한다. 이 조건이 기록되지 않은 경우에는 가상 세포 연구 의제의 구성 요소로만 기술한다.
{% endhint %}
