# 0. 노트북 실행 규칙

## 이 튜토리얼을 따라가는 법

이 파일(01.md)부터 [Chapter 10](README.md)의 실습 노트북이 시작됩니다. 대사모델링을 처음 접한다면, 셀을 실행하기 전에 아래 세 가지만 먼저 확인하면 한 단계도 막히지 않고 따라올 수 있습니다.

1. **준비물을 먼저 갖춥니다.** 이 튜토리얼은 [COBRApy](https://opencobra.github.io/cobrapy/) `0.30.0`을 격리된 가상환경(virtual environment)에 설치해 실행합니다. 이 장 전용 설치 명령은 바로 아래 §0.1에 정리해 두었고, 환경 구성을 더 자세히 알고 싶으면 [Chapter 11 §1](../chapter-11/01.md)이나 [설치 가이드](../installation.md)를 따르면 됩니다. 버전을 `0.30.0`으로 고정해야 이 장의 숫자가 그대로 재현됩니다.
2. **셀을 위에서 아래로 순서대로 실행합니다.** 이 장은 01.md부터 15.md까지 하나의 노트북 흐름으로 이어집니다. 앞 셀이 만든 변수(`model`, `results` 등)를 뒤 셀이 이어받으므로, 중간을 건너뛰면 `NameError`가 납니다. 순서가 왜 중요한지와 각 코드 블록에 붙은 배지(독립 실행·선행 셀 필요 등)의 뜻은 바로 아래 §0.0에서 설명합니다.
3. **출력이 예상과 다르면 먼저 이렇게 진단합니다.** 숫자가 이상하거나 `assert`가 실패하면, 가장 먼저 노트북 커널을 재시작(Restart Kernel)한 뒤 위에서부터 전부 다시 실행해 봅니다. 그래도 다르면 `pip show cobra`로 설치된 버전이 `0.30.0`인지, `import sys; print(sys.executable)`로 커널이 가상환경(`.venv`)을 가리키는지 확인합니다.

## 0.0 실행 규칙부터 정해야 하는 이유

이 장은 서술형 챕터가 아니라 **실행 가능한 노트북**입니다. 실행 가능한 노트북에는 서술형 문서에는 없는 특유의 함정이 있습니다. 셀을 하나 건너뛰고 실행하면 `NameError: name 'model' is not defined`가 나고, 셀을 두 번 실행하면 이미 바뀐 상태 위에 또 변경을 쌓아 값이 달라질 수 있습니다. Jupyter는 파일에 적힌 순서가 아니라 **사용자가 실제로 실행한 순서**로 상태를 유지하기 때문입니다.

일상 비유를 들면, 이 장은 요리 레시피와 비슷합니다. "2단계: 양파를 볶는다"는 "1단계: 양파를 썬다"를 먼저 실행했다는 전제 위에서만 의미가 있습니다. 순서를 건너뛰거나 뒤바꾸면 레시피가 실패하듯, 이 장의 코드 셀도 앞선 셀이 만든 변수(`model`, `results`, `wt_growth` 등)에 의존합니다. 그래서 각 코드 블록 위에 그 셀이 무엇을 전제하는지 알려 주는 배지를 붙였습니다.

| 배지 | 의미 |
|:---|:---|
| **독립 실행** | 필요한 import와 입력을 셀 안에서 준비하므로 단독 실행 가능 |
| **선행 셀 필요** | 이 장의 앞선 셀에서 만든 `model`, `results` 등의 변수를 사용 |
| **외부 자산 필요** | 사용자가 준비한 SBML 같은 로컬 파일이 있어야 실행 가능 |
| **의사코드** | 알고리듬의 생각 순서를 보여 주는 설명이며 실행 대상이 아님 |

핵심 경로는 **‘독립 실행’과 ‘선행 셀 필요’ 셀을 위에서 아래로 실행**하는 것입니다. ‘외부 자산 필요’ 셀은 자신의 모델로 확장할 때만, ‘의사코드’ 블록은 구현 전에 논리를 확인할 때만 사용합니다.

{% hint style="warning" %}
**해석상의 주의: "노트북 커널을 재시작하지 않고 그냥 다시 실행하면 되겠지"**

Jupyter 커널을 재시작하지 않은 채 앞 셀을 고쳐서 다시 실행하면, 이미 메모리에 남아 있는 이전 상태(예: 이전 실험에서 바뀐 `model.medium`)가 새 실행에 섞여 들어갈 수 있습니다. 이 장의 §4처럼 `with model:` context manager로 감싸지 않은 임시 실험을 했다면 특히 위험합니다. 재현성을 의심할 상황(숫자가 이상하다, assert가 실패한다)에서는 **커널 재시작(Restart Kernel) 뒤 위에서부터 전부 다시 실행**하는 것이 가장 먼저 시도할 진단입니다.
{% endhint %}

## 0.1 한 번만 설치하기

가상환경(virtual environment)을 먼저 만드는 이유는 간단합니다. 컴퓨터에 이미 다른 프로젝트용 파이썬 패키지가 깔려 있을 수 있고, 그 버전이 이 장이 요구하는 `cobra==0.30.0`과 충돌할 수 있기 때문입니다. `.venv` 폴더 하나에 이 장 전용 패키지 집합을 격리해 두면, 다른 프로젝트를 건드리지 않고도 정확한 버전으로 모든 숫자를 재현할 수 있습니다.

> **독립 실행 · 터미널**

```bash
python3 -m venv .venv
source .venv/bin/activate              # Windows PowerShell: .venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install "cobra==0.30.0" swiglpk numpy pandas \
  jupyterlab ipykernel plotly ipywidgets
python -m pip check
```

- 1번째 줄: 현재 폴더 아래 `.venv`라는 이름으로 독립된 파이썬 환경을 만듭니다.
- 2번째 줄: 그 환경을 "활성화"합니다. 활성화 후 프롬프트 앞에 `(.venv)`가 붙으면 성공입니다.
- 4~5번째 줄: `cobra`는 버전을 정확히 `0.30.0`으로 고정하고, `swiglpk`는 [GLPK](https://www.gnu.org/software/glpk/) solver의 파이썬 바인딩, `numpy`/`pandas`는 수치·표 계산, [`jupyterlab`](https://jupyter.org/)/`ipykernel`은 노트북 실행 환경, [`plotly`](https://plotly.com/python/)/[`ipywidgets`](https://ipywidgets.readthedocs.io/)는 §10~11의 대화형 그림에 필요합니다.
- 6번째 줄: `pip check`는 설치된 패키지들의 의존성 버전이 서로 충돌하지 않는지 확인합니다.

`python -m pip` 형태를 쓰면 지금 선택한 Python에 설치한다는 뜻이 분명해집니다. 설치 후 Jupyter 커널의 `sys.executable`이 `.venv` 안을 가리키는지도 확인하십시오.

```python
# 기대 출력: Jupyter 노트북 셀에서 실행
import sys
print(sys.executable)
# 기대 출력 예시: /path/to/project/.venv/bin/python
```

{% hint style="warning" %}
**흔한 에러와 해결**

- `ModuleNotFoundError: No module named 'cobra'` — Jupyter 커널이 `.venv`가 아닌 시스템 파이썬을 쓰고 있다는 뜻입니다. JupyterLab에서 커널을 `.venv` 안의 것으로 바꾸거나, `python -m ipykernel install --user --name cobrapy-tutorial`로 커널을 새로 등록하십시오.
- `swiglpk` 설치 실패(컴파일 오류) — macOS/Linux에서는 대개 pip이 미리 빌드된 wheel을 받아 오지만, 드물게 컴파일러가 필요한 환경도 있습니다. `pip install --upgrade pip setuptools wheel`을 먼저 실행한 뒤 다시 시도하십시오.
- `AssertionError: 이 장의 기준 버전은 0.30.0입니다` (§1에서 발생) — `pip show cobra`로 실제 설치된 버전을 확인하고, `pip install "cobra==0.30.0" --force-reinstall`로 맞추십시오.
{% endhint %}

{% hint style="info" %}
`load_model("textbook")`은 처음 실행할 때 원격 저장소에서 모델을 내려받아 로컬 캐시에 보관할 수 있습니다. 첫 실행에는 네트워크가 필요할 수 있으며, 엄밀한 재현이 필요하면 §12처럼 실제 분석 모델을 [SBML](../supplements/sbml-practical.md)로 저장하고 checksum을 함께 기록하십시오. 오프라인 환경이라면 미리 한 번 네트워크가 되는 곳에서 `load_model("textbook")`을 실행해 로컬 캐시를 만들어 둔 뒤 옮기는 방법도 있습니다.
{% endhint %}

## 0.2 대안 solver로 확장하기(선택)

이 장은 위 설치만으로 충분한 무료·오픈소스 GLPK 하나만으로 끝까지 실행됩니다. 하지만 실무에서 규모가 큰 모델의 [MILP](09.md)를 풀거나(§8), 원형(비선형) MOMA처럼 QP(quadratic programming) solver가 필요한 계산을 하려면([§7](08.md)의 힌트 참고) 상용 solver가 필요할 수 있습니다. [Gurobi](https://www.gurobi.com/)는 학술 이메일로 무료 학술 라이선스를 받을 수 있는 대표적인 QP·MILP solver입니다. 설치·라이선스 발급 절차는 이 장에서 반복하지 않고 [설치 가이드](../installation.md)에 정리해 두었으니, GLPK로 한계를 느낄 때 그 문서를 따라가면 됩니다. solver를 바꾼 뒤에는 §1.1처럼 `configuration.solver`와 `model.solver.interface.__name__`을 다시 확인해, 의도한 solver가 실제로 쓰이는지 검증하는 습관을 들이십시오.
