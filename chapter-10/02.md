# 1. Preflight: 버전·solver·모델을 먼저 고정하기

## 1.0 계산보다 환경 확인이 먼저인 이유

비행기 조종사가 이륙 전에 "preflight checklist"를 점검하듯, 계산 노트북도 첫 계산을 하기 전에 "이 결과가 어떤 환경에서 나왔는가"를 먼저 고정해야 합니다. 그 이유는 단순합니다. **[COBRApy](https://opencobra.github.io/cobrapy/) 버전, solver 종류, 모델 파일이 다르면 같은 코드에서도 다른 숫자가 나올 수 있기 때문**입니다. 예를 들어 solver가 GLPK에서 CPLEX로 바뀌면 동일한 LP라도 대안 최적해 중 어떤 것을 반환하는지가 달라질 수 있고, `cobra` 버전이 바뀌면 함수 이름이나 반환값의 자료구조가 바뀔 수 있습니다.

이 절의 코드는 이 세 가지를 실행 초반에 못박아, 이후 §2~§13에서 나오는 모든 숫자가 "이 정확한 조합에서" 나온 것임을 보증합니다.

## 1.1 시작 셀

> **독립 실행**

```python
import sys
import numpy as np
import pandas as pd
import cobra

from cobra import Configuration
from cobra.io import load_model

# cobra 버전이 다르면 이 장의 모든 기준값(assert)이 깨질 수 있으므로 가장 먼저 확인.
assert cobra.__version__ == "0.30.0", (
    f"이 장의 기준 버전은 0.30.0입니다. 현재 버전: {cobra.__version__}"
)

# Configuration: cobra 전역 설정을 관리하는 싱글턴(singleton).
# 여기서 지정한 solver·processes는 이후 새로 만드는 모든 모델의 기본값.
configuration = Configuration()
configuration.solver = "glpk"      # solver를 명시적으로 고정해 "기본값이 우연히 다른 solver"인 상황을 방지.
configuration.processes = 1        # 교육용 예제는 단일 프로세스로 실행 순서와 출력을 재현.

model = load_model("textbook")     # BiGG의 e_coli_core를 COBRApy 내장 별칭 "textbook"으로 적재.
active_solver = model.solver.interface.__name__

# 아래 세 assert는 "내가 생각하는 모델"과 "실제로 불러온 모델"이 같은지 검증하는 안전장치.
assert active_solver.endswith("glpk_interface")
assert (len(model.reactions), len(model.metabolites), len(model.genes)) == (95, 72, 137)
assert model.reactions.has_id("Biomass_Ecoli_core")

# 뒤의 셀에서 숫자와 provenance를 모을 사전. 이 장 전체에서 하나의 results만 사용.
results = {}

print("Python:", sys.version.split()[0])
print("COBRApy:", cobra.__version__)
print("solver:", active_solver)
print("model:", model.id)
print("reactions / metabolites / genes:",
      len(model.reactions), len(model.metabolites), len(model.genes))
```

```text
# 기대 출력:
Python: 3.11.x        (설치된 파이썬 버전에 따라 달라짐)
COBRApy: 0.30.0
solver: glpk_interface
model: e_coli_core
reactions / metabolites / genes: 95 72 137
```

기준 환경에서는 모델 ID [`e_coli_core`](http://bigg.ucsd.edu/models/e_coli_core), 반응 95개, 대사물 72개, 유전자 137개가 출력됩니다. `processes=1`은 수치해를 바꾸기 위한 설정이 아니라, 작은 교육 예제에서 운영체제별 multiprocessing 차이를 줄이기 위한 설정입니다. `e_coli_core`의 GPR·구획·바이오매스 반응 구조 자체는 [Chapter 3](../chapter-3/README.md)에서 자세히 다루었고, 이 모델로 FBA를 처음 실행하는 절차는 [Chapter 4](../chapter-4/README.md)에서 이미 살펴보았습니다. 여기서는 그 지식을 전제로 곧바로 실행 흐름을 이어 붙입니다.

{% hint style="warning" %}
**흔한 에러와 해결**

- `AssertionError: 이 장의 기준 버전은 0.30.0입니다` — §0.1의 안내대로 `pip install "cobra==0.30.0" --force-reinstall`로 버전을 맞추십시오.
- `active_solver`가 `"glpk_interface"`로 끝나지 않음(예: `"cplex_interface"`) — 다른 solver가 설치되어 자동 선택되었을 수 있습니다. `configuration.solver = "glpk"`를 `load_model` **이전에** 실행했는지 확인하십시오. Configuration은 전역 상태이므로 실행 순서가 중요합니다.
- 반응/대사물/유전자 개수가 `(95, 72, 137)`과 다름 — `load_model("textbook")`이 아니라 다른 모델(예: 로컬에 캐시된 오래된 버전)을 불러왔을 가능성이 있습니다. `cobra.io.load_model`의 캐시 경로를 확인하거나 캐시를 지우고 다시 시도하십시오.
- 인터넷이 없는 환경에서 `load_model("textbook")`이 멈추거나 실패 — §0.1의 힌트처럼 네트워크가 되는 곳에서 먼저 한 번 실행해 로컬 캐시를 만들어 두거나, §1.2처럼 이미 갖고 있는 SBML 파일을 `read_sbml_model`로 불러오십시오.
{% endhint %}

{% hint style="success" %}
**잠깐, 생각해보기**: `configuration.processes = 1`을 지우고 기본값(코어 개수만큼 병렬)으로 두면 이 절의 `assert`들이 실패할까요?

풀이 방향: 이 셀의 `assert`가 검사하는 대상(버전 문자열, solver 인터페이스 이름, 반응·대사물·유전자 개수, 반응 ID의 존재)을 하나씩 적어 보고, 그 각각이 LP를 몇 개의 프로세스로 나누어 푸는지에 따라 달라질 수 있는 값인지 따져 봅니다. `processes`가 실제로 관여하는 계산이 이 장의 어느 절에 나오는지도 함께 확인합니다. 답안은 [§14 말미의 답안 절](15.md)에 있습니다.
{% endhint %}

## 1.2 내 SBML 모델로 바꾸기(선택)

이 장의 나머지 절은 모두 `e_coli_core`를 기준으로 진행하지만, 실무에서는 결국 자신의 연구 대상 모델(효모, 특정 세균, 인체 조직 특이적 모델 등)로 같은 흐름을 반복해야 합니다. 이 절은 그 전환에서 가장 먼저 부딪히는 파일 입출력 단계만 미리 보여 줍니다.

> **외부 자산 필요 · 핵심 순차 실습에는 포함하지 않음**

```python
from pathlib import Path
from cobra.io import read_sbml_model

model_path = Path("models/my_model.xml")
assert model_path.exists(), f"파일을 찾을 수 없습니다: {model_path}"
user_model = read_sbml_model(model_path)
print(user_model.id, len(user_model.reactions))
```

- `Path("models/my_model.xml")`: 실제 파일 경로로 바꿔야 하는 자리입니다. 상대 경로는 노트북을 실행하는 현재 작업 디렉터리 기준입니다.
- `assert model_path.exists()`: 파일이 없는데 `read_sbml_model`을 호출하면 이해하기 어려운 XML 파서 오류가 나므로, 미리 명확한 메시지로 실패시킵니다.
- `read_sbml_model`: [SBML](https://sbml.org/)(Systems Biology Markup Language) 파일을 COBRApy `Model` 객체로 파싱합니다. SBML 형식 자체와 FBC(Flux Balance Constraints) 플러그인이 목적함수·bound를 어떻게 담는지는 [SBML 실무 보충 자료](../supplements/sbml-practical.md)에서 자세히 다룹니다.

자체 모델을 쓰면 이 장의 반응 ID(`EX_glc__D_e`, `Biomass_Ecoli_core` 등)를 그대로 가정하면 안 됩니다. 먼저 exchange, biomass, compartment, GPR ID를 모델 문서와 함께 확인해야 합니다. 모델을 새로 만들거나 최근에 gap-filling한 상태라면 [Chapter 5](../chapter-5/README.md)의 재구축·QC 절차를 먼저 적용해 MEMOTE([Lieven et al., 2020](../landmark-papers.md)) 같은 도구로 기본 건전성을 확인한 뒤 이 장의 흐름에 태우는 것이 안전합니다.

{% hint style="warning" %}
**흔한 에러와 해결**

- `AssertionError: 파일을 찾을 수 없습니다` — 상대 경로 문제일 가능성이 큽니다. 노트북 안에서 `from pathlib import Path; print(Path.cwd())`로 현재 작업 디렉터리를 확인하십시오.
- `libsbml` 관련 `ImportError` — `read_sbml_model`은 내부적으로 `python-libsbml`을 사용합니다. `cobra`를 pip으로 설치하면 보통 함께 설치되지만, conda 환경을 섞어 쓴 경우 누락될 수 있습니다. `pip install python-libsbml`로 별도 설치하십시오.
- 반응 ID가 이 장의 예제와 겹치지 않는 오류(`KeyError: 'EX_glc__D_e'`) — 다른 명명 규칙(BiGG가 아닌 자체 ID 체계)을 쓰는 모델일 수 있습니다. `[r.id for r in user_model.exchanges][:10]`처럼 실제 exchange 반응 ID를 먼저 나열해 대응 관계를 파악하십시오.
{% endhint %}
